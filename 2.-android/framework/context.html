Context

Interface to global information about an application environment. This is an abstract class whose implementation is provided by the Android system. It allows access to application-specific resources and classes, as well as up-calls for application-level operations such as launching activities, broadcasting and receiving intents, etc.

애플리케이션 환경에 대한 전역 정보에 접근하는 인터페이스입니다. 이 클래스는 추상 클래스이며, 구현은 안드로이드 시스템에서 제공합니다. 이 클래스를 통해 애플리케이션별 리소스 및 클래스에 접근할 수 있을 뿐 아니라, 액티비티 실행, 인텐트 전송 및 수신 등 애플리케이션 수준의 작업에 대한 호출도 가능합니다.

  • Android Official Document -

직역하면 "맥락" 또는 "환경 정보".
애플리케이션의 현재 상태에 대한 정보를 담고 있는 인터페이스이자 리소스/시스템에 접근하기 위한 게이트 웨이

애플리케이션과 관련된 정보에 접근하고자 하거나 애플리케이션과 연관된 시스템 레벨의 함수를 호출하고자 할 때 사용됨.

Context의 역할

리소스 접근

strings.xml, colors.xml, 이미지(drawables), 레이아웃 파일 등 앱에 정의된 리소스를 가져올 때 사용

시스템 서비스 연동

NotificationManager, LocationManager, ClipboardManager 등 안드로이드 시스템 서비스를 가져올 때 사용

컴포넌트 간 통신

새로운 Activity를 시작하거나, Intent를 전달하고, Broadcasr를 발생시킬 때 사용

Context의 계층 구조 (Class Hierarchy)

context는 추상 클래스이며, 실제 구현은 안드로이드 시스템에서 제공

Conetext(Abstract Class)

최상위 추상 클래스

ContextImpl

Context의 추상 메서드들을 실제로 구현한 구현부. 핵심 클래스

ContextWrapper

ContextImpl 인스턴스를 내부에 보관하고, 모든 호출을 해당 인스턴스로 전달하는 래퍼 클래스

Application / Activity / Service

ContextWrapper를 상속받아 구현된 컴포넌트들입니다. 즉, 하나의 Context입니다.

Application Context

획득 방법

호출함수 getApplicationContext()를 통해 사용 가능

생명주기

싱글톤 객체로 애플리케이션의 생명주기를 따름.
즉, 앱의 시작부터 종료까지(앱 프로세스가 살아있는 동안) 유지

사용목적

현재 Component가 아닌, Process의 수명에 연결된 현재 Context와 수명 주기가 분리된 Context가 필요한 경우에만 사용.
앱 전역에서 유지되어야 하는 싱글톤 객체을 위해 사용함.
(DB, 네트워크 라이브러리 초기화 등)

주의 사항

UI 작업에 부적합

다이얼로그나 팝업 같은 UI 요소를 화면에 그리려면 시스템 서비스인 WindowManager에게 요청해야 함. 이때 시스템은 해당 요청이 유효한 화면(Activity) 위에서 발생하는지 확인하기 위한 윈도우 토큰을 요구함. Application Context는 특정 화면에 종속되지 않음. 즉, 윈도우 토큰이 없기 때문에 시스템은 이 Context가 보낸 UI 생성 요청을 거절하며 WindowManager.BadTokenException을 발생시킴.

앱 프로세스가 살아있는 동안 계속 남아있어 GC(Garbage Collector)가 수집하지 않음

Application Context가 붙잡고 있어 GC가 치우지 못하는 주변 객체들을 할당 해제하지 않는다면 메모리 누수가 발생할 수 있음.(WeakReference를 사용해 해결)

Activity Context

획득 방법

Activity 내부에서 this키워드를 사용 또는 Activity.this를 통해 사용

생명주기

Activity가 소멸(onDestroy)되면 함께 소멸됨.

사용목적

UI 작업(다이얼로그 표시, 레이아웃 인플레이션), 특정 화면 내 리소스 접근

주의 사항

외부에서 참조 시 메모리 누수

onDestroy가 호출되어 해당 Activity 객체를 메모리에서 지우려하나, 만약 싱글톤 또는 장기적으로 생존하는 객체 내부의 변수가 해당 Activity를 참조(Strong reference)하고 있다면, GC에 의해 수집되지 않게 됨. 즉, 화면에서는 액티비티가 사라졌지만, 메모리 상에는 존재하는 상태가 됨.

구성 변경(Configuration Change) 시의 휘발성

안드로이드 기기를 회전(가로/세로 전환)하거나 시스템 언어를 변경하면, 안드로이드 시스템은 현재 액티비티를 완전히 파괴하고 새로 생성함. 만약 특정 백그라운드 작접에 Activity Context를 전달해 두었다면, 구성 변경 후 기존 Context는 유효하지 않은 상태(Destroyed)가 됨.

Service Context

획득 방법

Service 내부에서 this키워드를 사용 또는 Service.this를 통해 사용.

생명주기

시작된 서비스 (startService)

stopSelf()가 호출되거나 시스템에 의해 강제 종료될 때까지 메모리에 유지됨.
액티비티가 화면에서 사라져도 Service Context는 살아있음.

바인딩된 서비스 (bindService)

해당 서비스에 바인딩된 모든 클라이언트(액티비티 등)가 연결을 해제(unbind)하면 Service Context가 소멸.

사용목적

UI가 없는 환경에서 시스템 리소스에 접근하거나 장기적인 작업을 수행할 때 사용
시스템 서비스 접근, 네트워크 및 DB 작업, Broadcast 발생 등

주의 사항

Service Context에서 수행되는 모든 작업은 메인(UI) 스레드에서 동작

Service Context를 사용하여 무거운 네트워크 작업이나 파일 I/O를 직접 수행하면 ANR 현상이 발생하므로, 별도의 스레드 또는 코루틴을 활용해야함.
