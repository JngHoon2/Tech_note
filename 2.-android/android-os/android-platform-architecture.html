Android Platform Architecture

Android는 다양한 기기와 폼 팩터에 사용할 수 있도록 제작된 Linux 기반의 오픈소스 소프트웨어 스택이며,계층적(Layer System)으로 구성되어 있음.

Layer System

하위 계층의 서비스를 상위 계층에게 제공함으로써, 상위 계층이 하위 계층의 서비스를 사용하는 방식으로 시스템을 구축하는 것. 계층적 구조를 통해 최종적으로 사용자가 사용하는 서비스(최상위 계층의 서비스)는 모든 계층의 서비스를 손 쉽게 사용할 수 있음.

하위 계층의 기능 구현 사항을 상위 계층이 모르게 하며, 하위 계층이 제공하는 인터페이스를 통해 사용

Linux Kernel

커널(kernal)은 운영체제의 핵심이 되는 컴퓨터 프로그램의 하나이며,시스템의 모든 것을 완전히 통제함. 운용체제의 다른 부분 및 응용 프로그램 수행이 필요한 여러 가지 서비스를 제공함. ART(Android RunTime)는 스레딩 및 low-level 메모리 관리와 같은 기본 기능에 Linux Kernel을 사용함.

Linux 커널을 사용하여 Android의 주요 보안 기능을 활용할 수 있으며, 기기 제조업체에서 커널용 하드웨어 드라이버를 개발할 수 있음.

하드웨어와 OS를 연결하는 매개존재로써, OS에 있어서 핵심

Power Management

가장 근원적인 관리를 포함하여 전원 종료, 비활성화, 하드웨어에 공급되는 전력에 대해 관리.

Hardware Driver

각종 하드웨어를 제어하는데 필요한 명령어 모음인 드라이버

Abstraction

하드웨어에 직접 접근하는 것은 복잡하며 보안상 취약이 존재할 수 있음.
때문에 일반적으로 커널은 운영체제의 복잡한 내부를 감추고 일관성 있는 인터페이스를 하드웨어에 제공하기 위해 같은 종류의 하드웨어에 대한 공통 명령어 집합인 추상화들로 이루어짐.

Resource Management

시스템 자원(CPU, Memory)을 효율적으로 관리해야 시스템을 최대한 원활하게 사용할 수 있음.
자원 관리에 해당하는 대표적인 기능인 CPU-Scheduling 포함.

Security

하드웨어와 프로세서의 보안을 담당.

Hardware Abstraction Layer(HAL)

복잡한 하드웨어 내부를 감추고 일관성 있는 인터페이스를 제공하기 위해 같은 종류의 하드웨어에 대한 공통 명령어 집합으로 묶는 과정을 수행하는 계층. 이를 통해 상위 계층인 Java API Framework 계층에 하드웨어 기능을 제공하는 표준 인터페이스를 제공.

하드웨어 공급업체에서 구현해야 하는 표준 인터페이스를 정의하며 안드로이드에서 하위 드라이버 구현을 고려하지 않아도 되게 해주는 시스템.

여러 러이브러리 모듈로 구성되어 있으며 특정 유형의 하드웨어 구성 요소를 위한 인터페이스를 구현함.
Framework API가 하드웨어에 액세스하기 위해 호출을 수행하면 Android 시스템이 해당 하드웨어 구성 요소에 대한 하이브러리 모듈을 로드해줌.

ART

Android RunTime

Android RunTime

Android RunTime, ART
Linux Kernel과 HAL의 상위 계층에 위치하여 운영체제를 활용하여 Java 애플리케이션 실행에 대한 중추 역할을 수행하는 VM(Virtual Machine)

ART의 등장 배경 및 역할을 이해하기 위해서는 JVM에 대해 이해하고 넘어갈 필요가 있음.

JVM

Java와 같은 high-level 언어는 컴퓨터가 이해할 수 있도록 변환(.java -> .class)해주는 과정인 컴파일과정을 거쳐야한다. 일반적인 컴파일 언어는 다이렉트로 기계어로 변환하지만, 자바 컴파일러는 기계어가 아닌 JVM이 이해할 수 있는 바이트 코드로 변환한다.(.class) 덕분에 자바 바이트 코드는 JVM만 설치되어 있으면 어떤 운영체제에서라도 실행될 수 있다.

안드로이드는 기본 언어로 Java를 채택했기 때문에 JVM이 필수적이 였으나 라이센스 및 효율 문제로 인해 안드로이드 운영체제 구조에 맞춰 구동할 수 있는 가상머신이 필요해짐. 이러한 배경으로 인해 Dalvik VM과 ART가 등장함.

Dalvik과 ART

ART는 Android 4.4(API 19)에서 등장하였으며, 해당 버전에서는 Dalvik과 함계 선택적으로 사용 가능했으나, Android 5.0(API 21) 이상에서는 ART가 기본 런타임 환경이 되었음.

Dalvik과 ART의 가장 큰 차이는 컴파일 방식이며, Dalvik은 JIT(Just-In-Time), ART는 AOT(Ahead-Of-Time)방식을 사용함.

JIT

  • 앱 실행 시 컴파일
  • 설치 시 컴파일을 하지 않기 때문에 AOT에 비해 설치 속도 빠름
  • 실행 시 컴파일을 하기 때문에 AOT에 비해 실행 속도 느림
  • 용량 작음

AOT

  • 앱 설치 시 컴파일
  • 설치 시 컴파일을 완료하기 때문에 JIT에 비해 설치 속도 느림
  • 실행 시 컴파일을 하지 않기 때문에 JIT에 비해 실행 속도 빠름
  • 용량 큼(미리 컴파일하여 가지고 있기 때문)

Dalvik

Dalvik Virtual Machine

Java는 컴파일러를 통해서 생성되는 바이트코드(.class)는 기계어(Machine Code)가 아님.
즉, 바이트코드는 특정 플랫폼에 종속된 것이 아닌, VM을 위한 코드일 뿐임. 따라서 인터프리터를 이용해서 기계어로 해석하는 과정이 필요함.

JIT

Just-In-Time

JIT 컴파일방식은 프로그램 실행 시 자주 사용되는 바이트코드(Hotspot)에 대해서는 미리 기계어로 해석해놓는 것(Caching)임. 때문에 바이트코드가 사용될 때 재해석할 필요가 없어 속도가 향상됨.

DVM의 JIT는 임계값을 초과하면 바이트코드를 기계어로 해석하는, 즉 어떤 구간이 특정 횟수 이상 반복되면 컴파일하는 Trace JIT 방식이다.

컴파일은 별도의 컴파일 스레드에 의해서 진행되며, 컴파일이 완료되면 Translation Cache에 저장함.

초기 DVM에는 JIT이 없었고 Android 2.2에서 적용됨. 이전까지는 애플리케이션 구동 시 실시간으로 CPU에 맞추어 코드를 변환했음. JIT 도입 후 코드의 일정 부분을 RAM에 올려두고 작업할 수 있기에 비약적인 성능 향상이 일어났으나, 다음과 같은 문제가 발생했음.

  • 컴파일러가 돌아가는 동안 배터리 및 하드웨어의 부하 발생
  • 애플리케이션 프로세스 전체를 RAM에 올려두고 사용해야함.

해당 문제를 해결하기 위해 AOT 컴파일 방식을 기반으로 만들어진 VM이 바로 ART.

ART

Android RunTime

Android 7.0 부터는 앱 설치시간을 단축하기 위해 최초 설치 시에는 JIT 컴파일 방식을 사용하고, 기기를 사용하지 않는 시간에 일부분 컴파일 작업을 실시하여 점진적으로 AOT 컴파일 방식으로 바꿔 나가도록 되어 있음. 즉, DVM과 ART의 장점을 합치고 단점을 해결하려는 시도.

AOT 컴파일러는 APK를 설치할 때 기계어로 해석함. 이때 사용되는 것은 APK에 포함된 Dalvik에서 사용하는 dex 파일.

Dalvik은 dexopt라는 툴을 통해서 dex파일을 최적화한 odex(optimized dex)라는 파일을 생성함.
odex는 특정 기기의 시스템에 최적화된 코드이기 때문에 다른 기기에서 사용할 수 없음.
DVM은 odex 파일을 앱 실행 시 기계어로 해석함.

ART는 AOT 컴파일 시 dex2oat라는 툴을 사용해서 dex파일을 odex로 변경한 후에 oat 파일로 변경함. oat 파일은 elf(executable file) 파일 형식의 기계어를 포함하는 파일임.

Native C/C++ Libraries

기본적으로 운영체제에 내장된 라이브러리들이 존재하는 계층.
모바일 기기는 하드웨어 성능이 PC에 비해 제한적이기에 효율적인 구동을 위해 low-level에 가까운 C/C++로 작성되어 있음.

해당 네이티브 라이브러리에 접근할 때는 Android NDK(Native Development Kit)을 사용함.

ART 및 HAL 등의 많은 핵심 Android 시스템 구성 요소 및 서비스가 C/C++ 네이티브 코드로 빌드되어 있으며, Java Framework API를 제공하여 이러한 일부 네이티브 라이브러리의 기능을 앱에 제공합니다.

Java API Framework

Android Framework를 의미하며 안드로이드 전체 기능은 Java로 작성된 API를 통해서 접근이 가능

개발자는 이를 통해 안드로이드 시스템 애플리케이션이 사용하는 것과 동일한 프레임워크 API에 대한 전체 접근 권한을 가지며 이를 바탕으로 다양한 기능을 개발할 수 있음.

개발자가 코드를 작성하는 애플리케이션 계층과 하위 계층들과 소통할 수 있도록 하는 브릿지 역할을 함.
액티비티 생명주기 및 언어 설정 등 프레임워크 수준의 기능을 담당.

System Application

최상위 계층.
일반 사용자들이 접하는 계층으로, 안드로이드 기본 제공 앱 및 스토어에서 다운받은 앱들이 해당 계층에 속함.
